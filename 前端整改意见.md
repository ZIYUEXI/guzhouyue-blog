# 孤舟月博客前端整改意见

## 一、评估结论

当前项目已经具备清晰的视觉风格和完整的首页展示结构，首屏品牌感较强，适合作为博客站点的视觉基础。

但从“可正式上线的博客网站”角度看，目前仍停留在静态首页原型阶段。主要问题是：用户能看到文章、分类、归档、搜索、专题和页脚入口，但多数入口无法完成真实阅读、检索、订阅或跳转动作。

整改重点应优先放在“内容可访问、文章可阅读、链接可用、搜索有效、SEO 可收录”这几件事上。

## 二、核心问题

### 1. 缺少真实文章阅读路径

首页已有文章卡片、搜索结果和归档条目，但这些内容目前不能进入真实文章详情页。

影响：

- 用户无法完成从首页进入文章正文的核心路径。
- 搜索、分类、归档只能展示，不能形成有效阅读闭环。
- 没有独立文章 URL，不利于分享、收藏和搜索引擎收录。

整改建议：

- 为每篇文章增加独立详情页。
- 首页文章卡片、归档条目、搜索结果都应链接到对应文章。
- 文章页应包含标题、发布时间、分类、阅读时长、正文、上一篇/下一篇。

优先级：高

## 2. 内容数据写死在组件中

文章、分类、归档数据目前集中写在 `src/App.tsx` 中。

影响：

- 每次发文都要改页面代码。
- 内容维护和页面组件耦合，后续文章数量增加后难以管理。
- 不利于引入 Markdown、RSS、sitemap、搜索索引等能力。

整改建议：

- 先将文章、分类、归档数据拆到独立数据文件。
- 中期引入 Markdown 或 MDX 内容系统。
- 为文章建立统一 frontmatter 规范，例如：

```yaml
title: 月下独行：关于慢写作的几个念头
description: 不是所有表达都需要赶路，有些文字适合在夜里慢慢抵达。
date: 2026-05-15
category: 人间札记
tags:
  - 写作
  - 生活
readingTime: 6 分钟读完
slug: slow-writing-under-moon
```

优先级：高

## 3. 搜索功能不完整

当前搜索只匹配标题、摘要和分类。快捷搜索中存在“最新文章”等词，但文章数据里并不包含这个关键词，点击后会出现空结果。同时搜索结果不可点击，也没有空状态提示。

影响：

- 搜索看起来可用，但实际检索能力弱。
- 用户点击快捷词后可能得到空白反馈。
- 搜索无法帮助用户进入文章。

整改建议：

- 搜索结果应链接到文章详情页。
- 增加“无结果”状态提示。
- 快捷搜索词应改为真实可匹配的分类、标签或关键词。
- 后续支持标题、摘要、正文、分类、标签一起检索。

优先级：高

## 4. 页脚存在无效链接

页脚中的 `RSS`、`GitHub`、`站点地图`、`隐私说明` 当前都是 hash 占位链接，没有对应页面或真实地址。

影响：

- 用户点击后不会到达有效内容。
- 会降低站点可信度。
- RSS 和 sitemap 对博客站尤其重要，缺失会影响订阅和收录。

整改建议：

- 没有准备好的链接不要作为正式入口展示。
- RSS 应指向 `/rss.xml`。
- 站点地图应指向 `/sitemap.xml`。
- GitHub 应指向真实仓库或作者主页。
- 隐私说明应建立独立页面，或暂时移除入口。

优先级：高

## 5. SEO 基础不足

当前只有首页 `title` 和 `description`，缺少文章级 SEO 信息。

影响：

- 搜索引擎无法按文章维度收录。
- 分享到社交平台时缺少标题、描述和封面图。
- 博客长期内容沉淀能力不足。

整改建议：

- 增加每篇文章的独立 meta title 和 description。
- 增加 canonical 链接。
- 增加 Open Graph 和 Twitter Card 信息。
- 增加 JSON-LD Article 结构化数据。
- 自动生成 `sitemap.xml`、`robots.txt` 和 `rss.xml`。

优先级：高

## 6. 可访问性仍需补强

搜索弹层缺少 Escape 关闭、焦点管理和关闭后焦点恢复。移动端菜单按钮缺少 `aria-expanded`，且打开后仍使用“打开菜单”的语义。

影响：

- 键盘用户和屏幕阅读器用户体验不完整。
- 后续做自动化可访问性测试时容易暴露问题。

整改建议：

- 搜索弹层支持 `Esc` 关闭。
- 打开搜索后焦点进入输入框。
- 关闭搜索后焦点回到搜索按钮。
- 搜索输入框增加明确 `aria-label`。
- 移动菜单按钮增加 `aria-expanded` 和 `aria-controls`。
- 菜单打开时按钮语义切换为“关闭菜单”。

优先级：中高

## 7. 缺少质量保障入口

当前 `package.json` 只有 `dev`、`build`、`preview`，没有测试、格式化、Lint 或 E2E 脚本。

影响：

- 只能确认是否能构建，不能防止交互和样式回归。
- 后续功能增加后，修改风险会明显上升。

整改建议：

- 增加 `lint` 和 `format`。
- 增加组件测试，覆盖搜索、主题切换、归档展开等逻辑。
- 增加 Playwright 冒烟测试，覆盖首页、移动菜单、搜索弹层和主要链接。

优先级：中

## 三、推荐整改顺序

### 第一阶段：补齐博客核心可用性

目标：让用户能真正读文章，站点入口不再是空壳。

任务：

1. 将文章数据从 `src/App.tsx` 拆出。
2. 增加文章详情页。
3. 让首页文章卡片、归档、搜索结果跳转到文章详情。
4. 修正页脚无效链接。
5. 搜索增加空状态和可点击结果。

验收标准：

- 点击任意文章卡片可以进入文章正文。
- 搜索结果可以进入对应文章。
- 页脚没有无效占位链接。
- 构建通过。

### 第二阶段：完善 SEO 和内容发布能力

目标：让博客具备长期内容沉淀和搜索收录能力。

任务：

1. 引入 Markdown 或 MDX。
2. 建立文章 frontmatter 规范。
3. 生成文章级 meta 信息。
4. 生成 `rss.xml`、`sitemap.xml`、`robots.txt`。
5. 增加分类页和归档页。

验收标准：

- 每篇文章都有独立 URL。
- 每篇文章都有独立标题、描述和发布时间。
- RSS 和 sitemap 可访问。
- 分类和归档能真实筛选文章。

### 第三阶段：补齐体验和工程质量

目标：降低后续维护成本，减少回归问题。

任务：

1. 修复搜索弹层和移动菜单可访问性。
2. 增加基础单元测试。
3. 增加 Playwright 冒烟测试。
4. 增加 ESLint、Prettier 或同类质量工具。
5. 优化首图体积，增加 WebP 或 AVIF 版本。

验收标准：

- `npm run build` 通过。
- 新增测试脚本通过。
- 桌面端和移动端主要页面无明显布局溢出。
- 搜索、菜单、主题切换交互正常。

## 四、建议优先级清单

| 优先级 | 整改项 | 收益 | 成本 |
| --- | --- | --- | --- |
| P0 | 增加文章详情页和真实阅读路径 | 解决博客核心不可用问题 | 中 |
| P0 | 修正文章、搜索、归档的跳转能力 | 建立完整阅读闭环 | 中 |
| P1 | 拆分内容数据 | 降低维护成本 | 低 |
| P1 | 修复页脚无效链接 | 提升可信度 | 低 |
| P1 | 增加 SEO、RSS、sitemap | 提升收录和传播 | 中 |
| P2 | 搜索空状态和正文检索 | 提升检索体验 | 中 |
| P2 | 可访问性修复 | 提升交互质量 | 中 |
| P2 | 增加测试和质量脚本 | 降低回归风险 | 中 |
| P3 | 图片格式和性能优化 | 改善首屏加载 | 低 |
| P3 | 评论、订阅、相关文章 | 提升留存 | 中 |

## 五、最终建议

不要先继续堆视觉模块。当前视觉已经足够支撑第一版博客，下一步应优先把“读文章”这条主链路打通。

推荐从以下三件事开始：

1. 拆出文章数据。
2. 增加文章详情页。
3. 修正所有能看见但不能真正使用的入口。

完成这三项后，再推进 Markdown、SEO、RSS、sitemap 和自动化测试。这样项目会从“展示型首页”转为真正可运营的博客网站。
